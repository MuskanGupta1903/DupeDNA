# DupeDNA Design System — "Diagnostic Lab Report"

This app must feel like reading a diagnostic lab test report — not a beauty
catalog, not a generic SaaS dashboard. Every UI decision should be checked
against this file before being written. If a component looks like a
"card-based SaaS dashboard," it is wrong — redo it.

## Colors
Define these as custom Tailwind colors via CSS variables. NEVER use default
Tailwind palette colors (red-500, blue-600, gray-100, etc.) anywhere in this
project — that includes borders, hover states, focus rings, and disabled
states, not just backgrounds.

| Token   | Hex     | Use case                                             |
|---------|---------|-------------------------------------------------------|
| paper   | #F1F0EA | Page background                                       |
| ink     | #1B2430 | Primary text, borders, headlines                      |
| stamp   | #C23B2E | Match-verdict badges ONLY — the one accent, don't reuse |
| rule    | #8FA8C9 | Dashed dividers, faint ruled lines                    |
| pass    | #3F7150 | Shared-ingredient overlap blocks ONLY                 |
| muted   | #6B6F76 | Secondary/meta text                                   |

## Typography
Load via Google Fonts, define as Tailwind font families.

- `font-headline` → **Special Elite** (typewriter). Use ONLY for the hero
  question and the match-verdict badge text. Nowhere else — not headings,
  not buttons, not nav.
- `font-body` → **IBM Plex Sans**. All UI text, labels, buttons.
- `font-mono` → **IBM Plex Mono**. All data: ingredient names, prices,
  percentages, sample numbers, footer metadata.

## Layout — single scrolling page, no route change for search → results

1. **Letterhead header bar**: "DUPEDNA LABS" left, "SAMPLE NO. ####" right
   (increment/randomize per session). 2px bottom border in `ink`.
2. **Hero**: `font-headline` question "What are we testing today?", an
   underline-style input (bottom border only, no box — like a signature
   line), a dark filled button "RUN TEST →", dashed "· · · SCROLL FOR
   RESULTS · · ·" cue below it.
3. **Results section** (revealed on scroll after search): "TEST RESULTS"
   left, "N MATCHES FOUND" right. Result rows separated by dashed
   horizontal rules. Each row:
   - Two-digit index (01, 02...) in muted mono
   - Product name (body, medium weight) + price/savings line (mono, muted)
   - Circular match-% badge in `stamp` red, rotated. ALTERNATE rotation
     between -8deg and 6deg per result — do not make every badge the same
     angle, the imperfection is intentional.
   - "INGREDIENTS · X/Y SHARED" mono label
   - Horizontal strip of small rectangular blocks: filled `pass` green for
     shared ingredients, outlined muted grey for unshared. This strip is
     generated dynamically from the real shared/total ingredient count —
     never hardcode or decorate it.
   - "Why this match" button (lazy-loads LLM explanation on click) +
     external-link icon button to `productUrl`
4. **Footer**: "END OF REPORT" left, "GENERATED BY INGREDIENT EMBEDDING
   MODEL" right.

## Hard non-goals (check every component against these)
- No rounded-corner, soft-shadow "card" styling anywhere. Ruled lines,
  dashed dividers, flat surfaces only.
- No default Tailwind palette colors, ever.
- No decorative-only elements — the overlap strip, badge %, sample number
  etc. must all be driven by real data.
- Don't let any component silently drift toward a generic dashboard look
  when in doubt, re-read this file rather than defaulting to your training
  priors on "clean modern UI."
